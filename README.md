# Olympiad JBNU - Prompt Engineering 🧠✨

> 🎓 2024 전북대학교 동계 빅데이터 캠프  
> 🏆 프롬프트 엔지니어링 경진대회 **우수상 수상**

<br><br><br>

## 📌 프로젝트 소개
이 프로젝트는 대규모 언어 모델(LLM)의 응답 품질을 높이기 위해  
프롬프트 설계와 컨텍스트 기반 답변 생성을 실험한 프로젝트입니다.

엑셀 기반 문제 데이터를 활용하여 질문별 프롬프트를 구성하고,  
모델 응답을 자동으로 생성 및 저장하는 구조로 구현했습니다.

<br><br><br>

## 🎯 주요 목표
- 질문에 맞는 프롬프트 구조 설계
- 추가 문맥 활용을 통한 응답 품질 개선
- 결과 저장 및 비교를 통한 성능 분석

## 🧪 대회 당시 개발 과정 및 성능 개선 전략

### 📊 전체 실험 요약

| 구분 | 실험 내용 | 적용 방식 | 효과 |
|------|----------|----------|------|
| 모델 설정 | Temperature 튜닝 | 0.4 ~ 0.7 실험 | 정확도 및 일관성 개선 |
| 프롬프트 설계 | 시스템 프롬프트 구조화 | To do / Not to do / 페르소나 | 답변 품질 향상 |
| RAG 적용 | context 추가 | 뉴스, PDF, 기존 답변 활용 | 정보 보강 및 정확도 상승 |
| 코드 개선 | question → context 기반 | add_rag 개선 | 신뢰도 증가 |
| 전략 실험 | 심리적 유도 | 특정 문장 구조 적용 | 점수 대폭 상승 |

<br><br>

### ⚙️ 1. 모델 설정 (Temperature)

| 값 | 특징 | 결과 |
|----|------|------|
| 0.5 | 기본 설정 | 4700점 이상 |
| 0.4 | 일관성 및 정확도 강화 | 4800점 이상 |
| 0.7 | 창의성 증가 (일부 문제 개선) | 4850점 이상 |

> 문제 특성에 따라 최적값을 탐색하는 방식으로 성능 향상

<br><br>

### 🧠 2. 시스템 프롬프트 설계

| 요소 | 내용 |
|------|------|
| 페르소나 | LLM 전문가 역할 부여 |
| To do | 명확한 설명, 근거 포함 |
| Not to do | 모호한 답변, 불필요한 정보 금지 |
| 강조 전략 | 중요한 규칙 반복 |
| 지시 방식 | 구체적이고 직접적인 명령 |

> 단순 문장이 아닌 **모델 행동을 제어하는 구조 설계**

<br><br>

### 📚 3. RAG 기반 개선 (add_rag)

| 구분 | 방식 |
|------|------|
| 개선 전 | question만 사용 |
| 개선 후 | context + question 결합 |
| 활용 데이터 | 뉴스 / PDF / 기존 답변 |
| 핵심 전략 | 낮은 점수 문제 중심 context 추가 |

> 정보 밀도 증가 → 정확도 향상

<br><br>

### 🔧 4. 코드 개선

- question 중심 → **context 기반 구조로 변경**
- 단순 처리 → **정보 결합 구조로 확장**
- 결과 → **정확도 + 신뢰도 증가**

<br><br>

### 📈 5. 정확도 향상 전략

| 전략 | 설명 | 효과 |
|------|------|------|
| 심리적 유도 | 특정 문장 및 구조 적용 | 답변 품질 향상 |
| 강한 명령 | 모델 행동 제어 | 원하는 답 유도 |

| 구분 | 점수 변화 |
|------|----------|
| 기존 | 약 3000점 |
| 개선 후 | 4000점 이상 |
| 최종 | 약 4850점 |

> 약 **1000점 이상 성능 향상**

<br><br>

### 🏁 6. 최종 결과

| 조건 | 결과 |
|------|------|
| 일반 프롬프트 | 약 4850점 |
| 강한 명령 적용 | 성능 ↑ / 일반화 ↓ |

> 정확도 극대화에 초점을 둔 전략

<br><br><br>

## 🔍 핵심 인사이트

- 프롬프트는 문장이 아니라 **구조 설계**
- 정확도 = **프롬프트 + context + 모델 설정**
- 단순 RAG보다 **문제 맞춤형 context가 중요**
- 핵심은 모델을 사용하는 것이 아니라 **모델의 행동을 통제하는 것**

<br><br><br>

## 🔧 리팩토링 과정 및 결과 (2026.04.01 기준)

| 단계 | 핵심 변경 | 기대 효과 |
|------|----------|----------|
| 1차 | 실행 흐름 분리 (`main.py` 책임 축소, 데이터 로드 / 프롬프트 생성 / API 호출 / 결과 저장 분리) | 가독성, 재사용성 향상 |
| 2차 | 하드코딩 제거 및 설정 분리 (`config.py`, 시스템 프롬프트/RAG 외부화) | 유지보수성, 수정 편의성 향상 |
| 3차 | 시스템 프롬프트 개선 + RAG 블록화 (`knowledge_blocks.json`, 점수 기반으로 관련 블록만 선택) | 불필요한 문맥 감소, 구조적 RAG 적용 |
| 4차 | mock 모드 + 로깅 추가 (`USE_MOCK`, `logger.py`, 실행 메타데이터 저장) | 서버 없이도 실행/시연 가능, 디버깅 가능 |

### 핵심 변화
- 단일 스크립트 구조 → 모듈형 구조
- 긴 문자열 기반 RAG → 선택형 Knowledge Block 기반 RAG
- 대회 서버 의존 실행 → mock/API 이중 실행 구조
- 단순 결과 저장 → 로그 및 실행 메타데이터 기반 추적 구조

<br><br><br>

## 🔍 리팩토링 후 개선된 사항

- 대회 전용 API 의존 구조를 mock/API 이중 실행 구조로 개선
- 프롬프트를 역할 / 규칙 / 출력 스타일 중심으로 재구성
- 긴 설명형 context를 `knowledge_blocks.json` 기반의 선택형 RAG 구조로 변경
- 데이터 로드, 프롬프트 생성, 응답 처리, 결과 저장 로직을 분리하여 유지보수성 향상
- 실행 로그 및 RAG 선택 메타데이터를 저장하도록 개선

<br><br><br>

## 🚀 향후 추가 개선 방향

olympiad-jbnu-prompt-engineering/
